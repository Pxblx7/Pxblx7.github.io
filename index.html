Propuesta de Desacoplamiento: Web Off Zecore

Una visión para desacoplar el Web Client de Zecore, mejorando la escalabilidad, resiliencia y agilidad a través de una Arquitectura Orientada a Eventos.

Resumen Ejecutivo

El proyecto busca resolver los problemas causados por el fuerte acoplamiento entre el Web Client y Zecore. Este acoplamiento genera cuellos de botella, limita el crecimiento y hace que el sistema sea frágil.

La solución es adoptar una Arquitectura Orientada a Eventos (EDA), creando servicios más pequeños e independientes para mejorar la escalabilidad y la resiliencia.

Objetivos Clave

  • Mejorar Independencia: Permitir que la web opere sin depender 100% de Zecore en tiempo real.
  • Aumentar Escalabilidad: Soportar más usuarios y transacciones sin degradar el rendimiento.
  • Incrementar Resiliencia: Aislar fallos para que un problema en un servicio no afecte a todo el sistema.

El Problema: Acoplamiento Estrecho

La arquitectura actual presenta "puntos de dolor" críticos que afectan directamente al negocio y a la experiencia del usuario.

Cuellos de Botella

Las llamadas síncronas a Zecore limitan la capacidad total del sistema.

Punto Único de Falla

Si Zecore falla, funciones críticas de la web dejan de funcionar.

Inconsistencia de Datos

Las fallas a mitad de un proceso generan datos inconsistentes.

Acoplamiento de Procesos

El frontend tiene demasiada responsabilidad, lo que lo vuelve frágil.

La Solución: Arquitectura Orientada a Eventos

Pasar de un monolito acoplado a un ecosistema de microservicios independientes que se comunican de forma asíncrona.

Nuevos Servicios y Responsabilidades

Se propone la creación y refactorización de varios servicios para dividir las responsabilidades y fomentar la independencia.

Customer Service

Gestionará el ciclo de vida del cliente, su perfil y direcciones, de forma independiente a los "Leads" de Zecore.

Cart Service

Gestionará carritos de compra activos, promociones y productos, sin depender del "Quotation" de Zecore.

Orders Service (Nuevo)

Orquestará todo el flujo del pedido, desde el pago hasta el envío, como corazón del proceso de compra.

Inventory Service (Nuevo)

Proporcionará información de stock casi en tiempo real, manteniendo una caché local.

Zecore Connector (Nuevo)

Actuará como "Capa Anti-Corrupción", traduciendo eventos entre la plataforma y Zecore.

Estrategia de Sincronización de Datos

Se implementará un enfoque de sincronización bidireccional para mantener los datos consistentes de forma eventual.

Plataforma → Zecore

Patrón "Transactional Outbox"

  1. El servicio guarda un cambio localmente.
  2. En la misma transacción, guarda el evento en una tabla "outbox".
  3. Un proceso separado publica el evento de forma fiable.
  4. El Zecore Connector consume y traduce el evento.

Zecore → Plataforma

Captura de Datos de Cambios (CDC)

  1. Se usan "ganchos" (hooks) en Zecore para detectar cambios.
  2. Zecore emite un evento cuando un dato cambia.
  3. Los servicios de la plataforma se suscriben y actualizan sus cachés locales.

Principio Clave: Consistencia Eventual

Los datos no se actualizarán instantáneamente. Este es un sacrificio deliberado para ganar mayor disponibilidad y rendimiento.

Plan de Transición por Fases

La implementación se hará por fases para minimizar riesgos, asegurando una migración controlada y segura.

  1. Fase 1: Lanzamiento Oscuro (Shadowing)

    Desplegar los nuevos servicios en producción sin que los usuarios los usen. Se copia el tráfico real para probar su comportamiento y validar la lógica sin afectar al sistema actual.

  2. Fase 2: Migración de Lecturas

    Lanzar el nuevo Inventory Service y modificar la web para que lea el stock de este servicio en vez de Zecore, obteniendo una ganancia de rendimiento y resiliencia rápidamente.

  3. Fases Futuras: Migración Gradual

    Continuar migrando cada parte del proceso (Submit de compra, creación de carritos, etc.) de forma independiente, lanzando cambios pequeños y controlados.
